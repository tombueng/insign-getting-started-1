Webhook Relay
Webhook data passes through 3rd party servers. When inSign sends a webhook callback, the payload is routed through a public relay service that you do not control. The webhook payload may contain:
  • Session IDs and transaction IDs that could be used to query or manipulate signing sessions
  • Signer names, email addresses, and other personal data embedded in callback parameters
  • Document metadata including filenames and status information
  • Status changes that reveal the signing workflow and timing

The relay service operator can see, log, and store all of this data in plain text. Endpoints are typically public URLs that anyone with the link could monitor.

Acceptable for sandbox/demo use. For production data, use a self-hosted webhook endpoint (Val.town or custom).

The problem: browsers block cross-origin requests
Browser
blocked
inSign

This app runs in your browser (e.g. on GitHub Pages) but needs to talk to your inSign server on a different domain. Browsers enforce Cross-Origin Resource Sharing (CORS) and block these requests unless the server explicitly allows them.

Option A: Local CORS proxy (recommended)
Browser
Proxy
inSign

A tiny Node.js proxy runs on your machine, forwards requests to inSign, and adds the CORS headers the browser expects. No server configuration needed, credentials never leave localhost.

node cors-proxy.js

Proxy runs on http://localhost:9009. That's it.

Install Node.js LTS, then open PowerShell:

cd $env:USERPROFILE\Downloads
node cors-proxy.js
brew install node
node ~/Downloads/cors-proxy.js
sudo apt install nodejs
node ~/Downloads/cors-proxy.js

No Node.js install needed:

docker run --rm -p 9009:9009 -v ./cors-proxy.js:/app.js node:alpine node /app.js
Option B: Configure CORS on the inSign server

If you have access to the inSign server configuration, you can allow this app's origin directly. Two properties are needed:

cors.filteringenabled
Must be set to true to activate CORS header processing. Default: off
cors.alloworigin
Comma-separated list of allowed origins. The Access-Control-Allow-Origin header is set to the matching value. Default includes origins for Azure SAML login and the Word add-in
Add this app's origin to the list:

Be careful not to remove the existing default origins - they are required for Azure SAML and the Word add-in. Append your origin to the existing list.

When you don't need any of this

The inSign sandbox at sandbox.test.getinsign.show has relaxed CORS settings and works directly from any origin - no proxy or server configuration needed. That's why the CORS proxy option is hidden when the sandbox is selected.

For your own server: if it is on the same origin as this app, or if you have already set cors.filteringenabled=true with your origin in cors.alloworigin, then you can disable the CORS proxy toggle - no proxy needed.
